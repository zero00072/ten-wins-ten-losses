# 營陣綱目風格指南（Specification Style Guide）

> **書寫語言**：文件以**繁體中文**為主，技術術語或英文縮寫依標準慣例保留。

---

## 1. 文件命名與種類（Document Types）

### 1.1 各專案必備文件

每個專案的工作目錄（通常為 `docs/`）下**必須**包含以下四種文件。為確保 AI 工作流的「零配置（Zero Configuration）」與極致順暢，檔名**全面廢除專案前綴，採絕對固定檔名**：

| 中文正式稱謂 | 固定檔名               | 用途                       |
| :----------- | :--------------------- | :------------------------- |
| 營陣綱目     | `specification.md`     | 技術規格的唯一真實來源。   |
| 許願池       | `wishlist.md`          | 待討論的新構想與功能需求。 |
| 問對錄       | `questions.md`         | 助理向主公提出的釋清問題。 |
| 淬鋒劄記     | `refactoring_notes.md` | 命名重構與設計決策記錄。   |

> [!NOTE]
> **雙名制**：各文件具備「中文正式稱謂」與「固定英文檔名」兩套名稱。行文時，技術脈絡用 `` `filename.md` ``，故事敘事脈絡用《》書名號或無符號中文稱謂。例：「將《營陣綱目》（`specification.md`）送入許昌」。

> [!NOTE]
> 進度報告（`progress_report.md`）、TUI 預覽（`tui_preview.md`）等文件由各專案視需要自行建立，不強制必備。

---

## 2. 主標題格式（Document Titles）

### 2.1 營陣綱目主標題

格式：`# <專案名稱> <中文功能描述>（<英文副標題>）營陣綱目`

```markdown
# 建置腳本（Build Scripts）營陣綱目

# Validator 自動化驗證工具（Validator）營陣綱目

# 安裝程式（Installer）營陣綱目
```

- 中文為主，英文用**全形括號** `（）` 括注
- 末尾統一加上「營陣綱目」兩字
- 禁止半形括號 `()` 在主標題使用

---

## 3. 章節格式（Section Format）

### 3.1 章節編號

**營陣綱目**的所有章節**必須**帶章節號。淬鋒劄記同樣使用章節號。許願池與問對錄不強制。

```markdown
## 1. 專案簡介（Project Overview）

## 2. 核心技術（Core Technologies）

## 3. 介面設計（Interface Design）
```

- 一級章節（`##`）：阿拉伯數字 + 點，如 `1.`
- 二級章節（`###`）：小數點，如 `1.1`、`1.2`
- 許願池、問對錄、淬鋒劄記**同樣使用阿拉伯數字**章節號

### 3.2 章節標題的中英文格式

所有章節標題均以**中文在前，英文全形括號在後**的格式書寫：

```markdown
## 2. 產出物目錄結構規範（Artifact Directory Layout）

### 2.1 根目錄層級檔案（Root-Level Files）
```

- 使用**全形括號** `（）`
- 英文部分每個單字首字母大寫（Title Case）

### 3.3 標題中的 Emoji

**所有文件的標題一律不使用 Emoji**。
Emoji 僅允許出現在正文內容中：

- 任務勾選標記：`✅`、`- [x]`
- Bug 嚴重性標示：`🔴`、`🟡`、`🟢`
- 預先宣告的文件首居提示區塊 Emoji（💡、⚠️）

### 3.4 分隔線

在**營陣綱目**的各大章節之間使用 `---` 分隔線，讓章節視覺上更清晰：

```markdown
## 3. 介面設計（Interface Design）

…章節內容…

---

## 4. 資料結構（Data Structures）
```

---

## 4. 評論區格式（Review Thread Format）

營陣綱目中的文若（荀彧）評論與主公回覆，需遵守以下固定格式。

### 4.1 文若（荀彧）的評論（主動發起）

```markdown
> **針對這一點，文若認為**：
> [評論內容]
```

- 固定措辭：「**針對這一點，文若認為**：」。
- 單一評論串中可有多個觀察點，以阿拉伯數字條列。

### 4.2 奉孝（郭嘉）的回覆

```markdown
> > @文若，謝謝你的觀察，針對這一點，我們：[回覆內容]
```

- 固定用**二層引用**（`> >`）。
- 起手語固定為：`@文若，謝謝你的觀察，針對這一點，我們：`。
- `@文若` 後加**全形逗號**。

### 4.3 文若（荀彧）的結案

```markdown
> ✅ **文若結案**：[結案說明]
```

- `✅` 在**粗體外**（即 `✅ **文若結案**`，非 `**✅ 文若結案**`）。
- 使用**一層引用**（`>`），與文若評論同層。
- **注意**：此行為只能由文若（荀彧）執行！奉孝（郭嘉）切勿自行結案。

### 4.4 結案後的補充

結案後**不可開新議題**。如需附加非阻塞性的資訊，用以下格式：

```markdown
> ✅ **文若結案**：[結案說明]
>
> 💡 **補充觀察**：[註意事項或備忘資訊]
```

- 補充前綴固定為 `💡 **補充觀察**：`
- 結案前的追問使用 `> **⚠️ 文若追問**：`（僅在結案**前**使用）

### 4.5 評論串完整範例

```markdown
> **針對這一點，文若認為**：
> `initramfs` 沒有副檔名，建議在規格中明確說明。
>
> > @文若，謝謝你的觀察，針對這一點，我們：已補上「無副檔名」的說明。
>
> ✅ **文若結案**：歧義消除，此議題結案。
```

## 5. 任務清單格式（Checklist Format）

### 5.1 執行者任務清單

```markdown
## N. 任務派發清單（Task List for Executor）

- [ ] **1. 任務名稱**
  - [ ] 子步驟描述
  - [ ] 子步驟描述
- [ ] **2. 任務名稱**
```

- 頂層任務用 `**N. 名稱**`（`N.` 為阿拉伯數字加點）。
- 子步驟**加勾選框**（`- [ ]`），方便追蹤細部進度。
- 已完成項目改為 `- [x]`。

### 5.2 開發藍圖清單（Roadmap）

```markdown
- [ ] **Phase 0：名稱（Name）**
  - [ ] 子任務描述
- [ ] **Phase 1：名稱（Name）**
  - [ ] 子任務描述
```

- 頂層任務階段使用 `**Phase N：名稱（英文名）**`。
- 子任務同樣附勾選框。

## 6. 中英文書寫規範（Writing Conventions）

### 6.1 中英文之間加空格

中文文字與英文字母、數字之間**必須加一個半形空格**：

```text
✅ 使用 `--installroot` 時的參數
❌ 使用`--installroot`時的參數

✅ 讀取 Linux 版本
❌ 讀取Linux版本
```

- 英文與全形標點間**不加**空格（如 `筆記（Note）`，`（` 前不加空格）
- 中文標點（。，：）後不加空格
- 年份純數字可不加空格（如 `2026年`）
- 行內程式碼（`` `code` ``）視為一個完整單元，其兩側加空格

### 6.2 IT 術語與中英對照

IT 術語（`ESP`、`GPT`、`TUI`、`CLI`、`SquashFS`、`initramfs` 等）**直接用英文**，不強制附中文翻譯。

若文件中需要括注以避免人類讀者誤解，可於**首次出現**時加全形括號說明，後續直接用英文：

```markdown
初始 RAM 磁碟（Initramfs）……後續直接稱 initramfs。
```

### 6.3 括號的全形 / 半形選擇

| 場景               | 括號類型    | 範例                                  |
| :----------------- | :---------- | :------------------------------------ |
| 中文語境的補充說明 | 全形 `（）` | `核心版本（Kernel Version）`          |
| 行內英文短語、縮寫 | 半形 `()`   | `kver.toml (Kernel Version Manifest)` |
| Markdown 超連結    | 半形 `[]()` | `[討論](url)`                         |
| 標題中的英文副標題 | 全形 `（）` | `## 1. 專案簡介（Project Overview）`  |

## 7. 文件開頭提示區塊與主標題（Document Header & Titles）

### 7.1 問對錄主標題

所有專案的 `questions.md` 必須具備主標題：

```markdown
# <專案名稱> 主公問對錄（Questions for Architect）
```

### 7.2 營陣綱目開頭警告（開發尚未開始時）

```markdown
> **⚠️ 提示**：本營陣綱目尚在起草或確認階段，所有細節確認前請勿開始實作。
```

### 7.3 許願池開頭提示

```markdown
> 💡 **提示**：這裡是許願池與設計討論區。您可以將任何新構想、功能需求或待辦事項寫在這裡。討論確認後，將逐步合併到正式營陣綱目中。
```

### 7.4 問對錄開頭提示

```markdown
> 💡 **提示**：這裡是各軍師向主公（您）提出實作問題與釋清規格的地方。
> 問題解決並整合至營陣綱目後，此處將保留空白供下次使用。
```

## 8. GFM Alert 區塊（Alert Blocks）

使用 GitHub Flavored Markdown 的 Alert 語法強調關鍵資訊：

```markdown
> [!NOTE]
> 背景脈絡、補充說明。

> [!IMPORTANT]
> 不可跳過的步驟或 Breaking Change。

> [!WARNING]
> 潛在風險或已知缺陷。

> [!CAUTION]
> 高危險操作（如磁碟抹除、不可逆變更）。
```

**使用原則**：

- 在普通的設計說明中**不過度使用** Alert，以免失去強調效果
- `[!IMPORTANT]` 保留給影響跨文件或跨專案的 Breaking Change。
- `[!CAUTION]` 保留給**真實硬體操作**的高危步驟。

## 9. 表格使用原則（Table Usage）

以下情況**必須**用表格呈現，不得改用條列：

| 使用表格的場景              | 理由                       |
| --------------------------- | -------------------------- |
| 命名對照（舊名 → 新名）     | 對照性資訊二維比較最清楚。 |
| CLI 參數說明（多欄位）      | 多屬性說明不適合條列。     |
| 測試指令對照（指令 / 情境） | 讀者需要快速查找對應關係。 |
| 版本字串格式規範            | 範例與說明需並排。         |

以下情況**不必**使用表格，條列即可：

- 只有單一屬性的清單。
- 步驟性說明（有先後順序的操作）。

## 10. 淬鋒劄記格式（Refactoring Notes Format）

```markdown
# <專案名稱> — 淬鋒劄記（Refactoring Notes）

> 日期：YYYY-MM-DD
> 作者：<作者名稱>
> 狀態：[進行中 / 已完成 ✅]
>
> [一句話說明本次重構的目的。]

---

## 1. 命名變更對照表（Naming Changes）

### 1.1 <檔案名稱>

| 舊名稱 | 新名稱 | 類型 | 理由 |
| :----- | :----- | :--- | :--- |

## 2. 修復的邏輯錯誤（Logic Bug Fixes）

### 2.1 🔴 高嚴重性：<標題>

### 2.2 🟡 中嚴重性：<標題>

### 2.3 🟢 低嚴重性：<標題>

## 3. 受影響的檔案清單（Changed Files）

## 4. 驗證結果（Verification）
```

- 主標題統一為「淬鋒劄記（Refactoring Notes）」，不用「美化調整筆記」
- 章節用**阿拉伯數字**，不用中文數字「一、二、三」
- Bug 嚴重性必須用 🔴🟡🟢 三色標示
